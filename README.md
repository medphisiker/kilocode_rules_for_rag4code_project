# Организация работы с агентом

Я завершил создание структуры управления знаниями, оптимизированной специально для работы в Kilo Code.

### Итоговая архитектура знаний:

1.  **Автоматическое Ядро (`.kilocode/rules/`)** — загружается в мою память автоматически:
    *   [`index.md`](.kilocode/rules/index.md:1) — **Главная точка входа**. Содержит карту всех знаний и краткую инструкцию "как работать с этим проектом".
    *   [`tooling.md`](.kilocode/rules/tooling.md:1) — **Инструментальные правила**. Переносимые стандарты (ADR, Git, поведение агента), которые можно брать в другие проекты.
    *   [`project.md`](.kilocode/rules/project.md:1) — **Проектные правила**. Специфика стека `rag4code`, структура папок и технические ограничения.

2.  **Ленивая Память (`.kilocode/memory/`)** — я читаю эти файлы только при необходимости:
    *   [`projectBrief.md`](.kilocode/memory/projectBrief.md:1) — Цели и контекст продукта.
    *   [`activeContext.md`](.kilocode/memory/activeContext.md:1) — Текущий статус, фокус и задачи.
    *   [`progress.md`](.kilocode/memory/progress.md:1) — (Создан как заглушка) История решений.

3.  **Ленивые Навыки (`.kilocode/skills/`)** — специализированные инструкции:
    *   [`documentation.md`](.kilocode/skills/documentation.md:1) — **Первый навык**. Подробный алгоритм работы с PlantUML, Docker-рендерером и ADR.

### Почему это эффективно:
*   **Нет дублирования**: Каждое правило живет в одном месте.
*   **Экономия контекста**: Я не держу в памяти инструкции по рендерингу диаграмм, пока они мне не нужны.
*   **Ясные точки входа**: Благодаря `index.md` я всегда знаю, где искать информацию, если чего-то не понимаю.

**Как вам такая реализация? Если всё устраивает, мы можем считать задачу по настройке рабочего окружения завершенной.**
